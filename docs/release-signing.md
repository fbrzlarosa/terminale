# Release signing

`terminale` ships installers for Windows (`.msi`), macOS (`.pkg`), and Linux
(tarballs). To deliver them **without OS security warnings** we need code
signing on Windows and Apple Developer-ID signing + notarization on macOS.

Linux installers do not require signing.

The release pipeline (`.github/workflows/release.yml` generated by `cargo-dist`,
plus `.github/workflows/sign-and-notarize.yml`) automatically signs and
notarizes when the required repository secrets are configured. If a secret is
missing, the matching step is skipped and the artifact still ships, but with
an OS warning.

---

## Windows

### Option A — SignPath.io OSS program (free, recommended to start)

[SignPath.io](https://signpath.io/open-source) offers free code signing for
qualifying open-source projects. Application process:

1. Apply at https://signpath.io/open-source
2. Once approved, generate an API token under **Settings -> CI integration**
3. Add these secrets in GitHub:

   | Secret | Value |
   |---|---|
   | `SIGNPATH_API_TOKEN` | The API token from SignPath |
   | `SIGNPATH_ORGANIZATION_ID` | UUID shown in SignPath dashboard |

4. Configure a SignPath **project** named `terminale` with a signing policy
   named `release-signing` and an artifact configuration named `msi`.

### Option B — Self-managed EV / OV code-signing certificate

Buy a certificate from a CA (DigiCert, Sectigo, SSL.com, GlobalSign, ...).
EV (Extended Validation) is recommended because it bypasses Windows SmartScreen
reputation immediately. OV works but new releases trigger SmartScreen until
they accumulate enough installs.

Export the certificate as a PFX file, then add these secrets:

| Secret | Value |
|---|---|
| `WINDOWS_PFX_BASE64` | `base64 -w0 your-cert.pfx` output |
| `WINDOWS_PFX_PASSWORD` | The PFX export password |

The pipeline uses `signtool` with timestamping against
`http://timestamp.digicert.com` so the signature stays valid after the
certificate expires.

## macOS

You need an [Apple Developer Program](https://developer.apple.com/programs/)
membership (99 USD / year).

1. In Apple Developer portal, create:
   - **Developer ID Application** certificate (signs the binary)
   - **Developer ID Installer** certificate (signs the .pkg)
2. Export each certificate as a `.p12` from Keychain Access (include private key)
3. Generate an [app-specific password](https://account.apple.com/account/manage)
   for the Apple ID you'll use to submit to the notary service.
4. Add these secrets in GitHub:

   | Secret | Value |
   |---|---|
   | `APPLE_TEAM_ID` | 10-character team ID from Apple Developer portal |
   | `APPLE_DEVELOPER_ID_INSTALLER_CERT_BASE64` | `base64 -i installer.p12` |
   | `APPLE_DEVELOPER_ID_INSTALLER_CERT_PASSWORD` | Export password of installer .p12 |
   | `APPLE_NOTARY_APPLE_ID` | Apple ID email |
   | `APPLE_NOTARY_APP_PASSWORD` | App-specific password generated above |
   | `APPLE_KEYCHAIN_PASSWORD` | Any password used to create the CI keychain |

The pipeline:

1. Creates a temporary keychain on the runner
2. Imports the installer certificate
3. Re-signs every `.pkg` with `productsign`
4. Submits to Apple's notary service via `xcrun notarytool submit --wait`
5. Staples the notarization ticket onto the `.pkg`

After this, Gatekeeper accepts the installer with no warning.

## Verifying signing locally

Once a tagged release runs and signing succeeds, verify on each OS:

### Windows

```powershell
Get-AuthenticodeSignature .\terminale-v0.1.0.msi
# Status should be `Valid`
```

### macOS

```bash
pkgutil --check-signature terminale-v0.1.0.pkg
spctl --assess --verbose=4 --type install terminale-v0.1.0.pkg
xcrun stapler validate terminale-v0.1.0.pkg
# All three must report success and an Apple-issued chain
```

### Linux

No signing applied; verify SHA256 instead:

```bash
sha256sum -c SHA256SUMS
```

## Falling back gracefully

If the project has not yet acquired the certificates, the release still
ships installers. Users see one of these warnings:

- **Windows**: "Windows protected your PC" - click "More info -> Run anyway"
- **macOS**: "cannot be opened because the developer cannot be verified" -
  right-click the .pkg -> "Open" -> "Open" once to authorize the install
- **Linux**: no warning

Once the secrets above are set in the repo, the next tagged release ships
fully signed and the warnings disappear.
